## Description
Currently, the cascade deletion between [Parent Model] and [Child Model] relies entirely on SQLAlchemy's default ORM-level cascade (`cascade="all, delete"`).

When a [Parent Model] is deleted, SQLAlchemy issues a SELECT statement to find all associated [Child Model] records, loads them entirely into application memory (RAM), and issues individual DELETE statements for each one before finally deleting the parent.

## Impact
While this works for small datasets, it creates a severe performance bottleneck and N+1 querying issue for large datasets. If a parent record has thousands of associated child records, deleting the parent can cause the API endpoint to hang, spike application memory usage, and potentially lead to Out-Of-Memory (OOM) crashes.

## Steps to Reproduce (or Current Behavior)
1. Create a [Parent Model] with 10,000+ associated [Child Model] records.
2. Trigger the deletion endpoint for the parent.
3. Observe the database logs: SQLAlchemy issues a SELECT for the children, followed by 10,000 individual DELETE statements.

## Expected Behavior
Deleting a parent record should issue a single DELETE command to the database, allowing the database engine itself to efficiently wipe out the child records via a foreign key constraint, bypassing the Python application's memory entirely.

## Proposed Solution
We should shift the cascade responsibility from the ORM to the database level.

- **Database Level**: Add `ondelete="CASCADE"` to the Foreign Key constraint on the [Child Model].

```python
# Example
parent_id = Column(Integer, ForeignKey('parents.id', ondelete="CASCADE"))
```

- **ORM Level**: Add `passive_deletes=True` to the relationship definition on the [Parent Model]. This instructs SQLAlchemy to stop loading the children into memory and trust the database's foreign key constraint.

```python
# Example
children = relationship("Child", cascade="all, delete", passive_deletes=True)
```
